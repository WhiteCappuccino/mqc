# 2.7 Описание процесса разработки

В данном разделе необходимо произвести описание процесса разработки спроектированной информационной системы, рассмотреть структуру проекта, назначение основных файлов и модулей, а также описать организацию клиентской, серверной и аналитической частей приложения.

При разработке проекта Media Quality Control Platform была применена модульная архитектура с разделением на несколько самостоятельных компонентов. Клиентская часть реализована в виде одностраничного приложения на основе React и TypeScript. Серверная часть представляет собой REST API на основе NestJS. Для автоматического анализа медиаматериалов используется отдельный сервис на Python и FastAPI. Хранение структурированных данных выполняется в PostgreSQL, а хранение загруженных файлов - в объектном хранилище MinIO. Взаимодействие между клиентской и серверной частями осуществляется посредством HTTP-запросов, а взаимодействие backend с analyzer выполняется через внутренний HTTP API.

Рисунок 58 - Файл app.module.ts серверной части

Файл `app.module.ts` объединяет все функциональные модули backend-приложения: `AuthModule`, `UsersModule`, `ProfileModule`, `MediaModule`, `ModerationModule`, `AdminModule`, `ReportsModule`, `AnalyticsModule`, `NotificationsModule`, `StorageModule`, `AnalyzerModule`, `AuditModule`, `PrismaModule` и другие. Такое разделение позволяет изолировать бизнес-логику отдельных частей системы.

Модуль `auth` отвечает за регистрацию, авторизацию, выход из системы, подтверждение электронной почты и восстановление пароля. В файле `auth.controller.ts` описаны HTTP-маршруты авторизации, а в файле `auth.service.ts` реализована бизнес-логика: проверка уникальности email и username, хэширование пароля, генерация JWT-токена, создание токенов подтверждения email, запись событий в журнал аудита и сохранение истории попыток входа.

Модуль `users` отвечает за работу с учетными записями пользователей. Он используется для поиска пользователя по email, username или идентификатору, создания новых пользователей, получения списка пользователей и изменения их статуса или роли.

Модуль `profile` реализует функции личного кабинета пользователя: получение профиля, изменение ФИО, имени пользователя, настроек уведомлений, смену пароля и просмотр истории пользователя.

Модуль `media` является одним из ключевых модулей системы. В файле `media.controller.ts` описаны маршруты для получения списка медиаматериалов, просмотра детальной информации, загрузки файла, загрузки новой версии, запуска автоматической проверки и управления доступами. В файле `media.service.ts` реализованы загрузка файла или получение файла по ссылке, проверка формата, вычисление контрольной суммы, сохранение файла в MinIO, создание записи `MediaItem`, запуск анализа через analyzer, сохранение результата `QualityCheck`, сохранение нарушений и изменение статуса медиаматериала.

Модуль `moderation` отвечает за ручную модерацию материалов. Он предоставляет модератору очередь материалов, позволяет принимать решение `APPROVED`, `REJECTED` или `NEEDS_REVISION`, добавлять нарушения вручную и отмечать автоматические нарушения как ложные срабатывания.

Модуль `admin` реализует административные функции системы. Через него выполняется управление пользователями, ролями, статусами, критериями проверки, словарем нарушений, системными настройками, журналом аудита и статусами медиаматериалов.

Модуль `analytics` отвечает за получение аналитических данных по медиаматериалам, статусам, нарушениям и пользователям. Модуль `reports` формирует отчеты в форматах CSV, XLSX и PDF.

Модуль `notifications` отвечает за создание уведомлений внутри системы и отправку email-уведомлений. Модуль `mail` содержит логику отправки писем через SMTP. Если SMTP не настроен, уведомления сохраняются внутри системы.

Модуль `storage` отвечает за взаимодействие с объектным хранилищем MinIO. Он сохраняет загруженные файлы, формирует ключи хранения и предоставляет публичные ссылки для анализа и просмотра медиаматериалов.

Модуль `analyzer` содержит клиент для взаимодействия с сервисом автоматического анализа. Он отправляет данные медиаматериала в сервис analyzer и получает результат проверки.

Модуль `audit` отвечает за запись действий пользователя в журнал аудита. В журнал сохраняются события регистрации, авторизации, загрузки медиаматериала, запуска автоматической проверки, изменения статусов, административных действий и других значимых операций.

Далее рассмотрим сервис автоматического анализа медиаматериалов. Он располагается в директории `analyzer` и реализован на Python с использованием FastAPI. Структура сервиса представлена на рисунке 61.

Место для вставки изображения: скриншот директории `analyzer`.

Рисунок 61 - Структура директории analyzer

Основной файл сервиса анализа - `analyzer/app/main.py`, представленный на рисунке 62.

Место для вставки изображения: скриншот файла `analyzer/app/main.py`.

Рисунок 62 - Файл main.py сервиса analyzer

В файле `main.py` описаны маршруты `GET /health` и `POST /analyze`. Маршрут `GET /health` используется для проверки доступности сервиса. Маршрут `POST /analyze` принимает данные медиаматериала, выполняет общие проверки и проверки по типу файла, рассчитывает итоговую оценку качества, формирует список нарушений и возвращает рекомендацию: одобрить материал, отправить на ручную проверку или отклонить.

Для изображений analyzer проверяет разрешение, резкость, шум, яркость, контраст, целостность и соответствие формата. Для видео проверяются длительность, разрешение, битрейт, наличие черных кадров и размытых кадров. Для аудио выполняются проверки длительности, громкости, шума, тишины и разборчивости. Для текстовых файлов выполняются проверки извлекаемости текста, длины, запрещенной лексики, читаемости и структуры.

В директории `docs` располагается дополнительная документация проекта. Файл `api.md` содержит описание основных REST API-маршрутов. Файл `testing.md` содержит команды запуска тестов для backend, frontend и analyzer. Файл `check-templates.md` описывает шаблоны критериев проверки и словарь нарушений. Файл `algorithms-section.md` содержит описание алгоритмов работы ключевых модулей системы.

Для проверки корректности реализации в проекте используются тесты. В backend присутствуют unit-тесты для сервисов авторизации, избранного, медиаматериалов и guards, а также e2e-тест для проверки работоспособности API. В frontend присутствует тест формы авторизации. В analyzer присутствуют тесты health endpoint и контракта ответа анализа.
