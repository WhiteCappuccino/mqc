# 2.6 Настройка технологического обеспечения

Перед началом непосредственной разработки информационной системы необходимо произвести настройку технологического обеспечения, окружения разработки и интегрированной среды разработки Visual Studio Code.

Настройка технологического обеспечения необходима для комфортной работы при разработке создаваемого программного обеспечения, а также для формирования условий, при которых разрабатываемая информационная система будет работать и на других устройствах. Данный раздел посвящен описанию настройки технологического обеспечения, формированию структуры проекта, установке зависимостей и настройке сервисов, необходимых для запуска системы.

В качестве интегрированной среды разработки был выбран Visual Studio Code. Данный инструмент обладает широкими возможностями расширения через систему плагинов и подходит для разработки веб-приложений на TypeScript, серверных приложений на NestJS, сервисов на Python, а также для работы с Docker, Prisma и Git.

В рамках данного проекта в качестве основной платформы разработки для клиентской и серверной частей был использован Node.js. Node.js представляет собой программную платформу, построенную на движке V8, которая позволяет выполнять JavaScript и TypeScript-код вне браузера. Вместе с Node.js устанавливается менеджер пакетов npm, используемый для подключения библиотек и запуска сценариев разработки.

Для установки Node.js необходимо загрузить установщик с официального сайта и выполнить стандартную установку. Пример официального сайта Node.js представлен на рисунке 37.

После установки зависимостей в директориях `frontend` и `backend` появляется папка `node_modules`. Данная папка содержит все установленные библиотеки и их внутренние зависимости. В нее помещаются пакеты, которые были установлены с помощью команды `npm install`. Папка `node_modules` может иметь большой объем, поскольку включает не только явно подключенные библиотеки, но и зависимости этих библиотек.

Одним из основных файлов Node.js-приложения является `package.json`. В данном проекте отдельные файлы `package.json` находятся в директориях `frontend` и `backend`. Это конфигурационные файлы, которые содержат сведения о проекте:

- имя проекта;
- номер версии;
- список зависимостей;
- список зависимостей для разработки;
- перечень доступных команд `scripts`.

При установке библиотек автоматически создается также файл `package-lock.json`. Данный файл фиксирует точные версии всех установленных зависимостей, включая вложенные зависимости. Основное назначение `package-lock.json` - обеспечить одинаковую установку пакетов на разных компьютерах. Благодаря этому файл `package-lock.json` гарантирует, что при повторной установке проекта структура зависимостей останется неизменной, а приложение будет работать одинаково в разных окружениях.

Отдельным компонентом проекта является сервис автоматического анализа медиаконтента `analyzer`. Он реализован на Python с использованием FastAPI. Данный сервис отвечает за автоматическую проверку загруженных изображений, видео, аудио и текстовых файлов, а затем передает результаты проверки в серверную часть.

Для локальной настройки analyzer необходимо создать виртуальное окружение Python и установить зависимости:

- `cd analyzer`;
- `python3 -m venv .venv`;
- `. .venv/bin/activate`;
- `pip install -r requirements.txt`.

Для хранения данных системы был выбран PostgreSQL. PostgreSQL представляет собой объектно-реляционную систему управления базами данных с открытым исходным кодом. В базе данных информационной системы хранятся пользователи, роли, медиаматериалы, результаты проверок, нарушения, комментарии, решения модерации, уведомления, избранное, коллекции, системные настройки и журнал аудита.

В данном проекте PostgreSQL запускается через Docker Compose на основе образа `postgres:16-alpine`. Такой подход позволяет не выполнять ручную установку PostgreSQL на рабочем устройстве и быстрее разворачивать одинаковое окружение на разных компьютерах.

Параметры базы данных в Docker Compose:

- имя базы данных - `media_quality`;
- пользователь - `postgres`;
- пароль - `postgres`;
- внутренний порт контейнера - `5432`;
- внешний порт на хосте - `5434`.

Для хранения загруженных файлов используется MinIO. MinIO является S3-совместимым объектным хранилищем и применяется для сохранения медиаконтента отдельно от реляционной базы данных. Такой подход позволяет хранить в PostgreSQL только метаданные файла, а сами изображения, видео, аудио и документы размещать в отдельном хранилище.

В проекте MinIO доступен по адресу `http://localhost:9000`, а web-консоль MinIO доступна по адресу `http://localhost:9001`. Пример настроек сервиса MinIO в файле `docker-compose.yml` представлен на рисунке 46.
