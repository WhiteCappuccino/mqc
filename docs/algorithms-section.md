# 2.5 Алгоритм работы модулей

Следующим этапом в разработке информационной системы является разработка алгоритмов работы основных модулей системы.

Алгоритм - набор инструкций, описывающих порядок действий исполнителя для достижения некоторого результата. В рамках разрабатываемой информационной системы алгоритмы позволяют формализовать последовательность действий пользователя, клиентского приложения, серверной части, базы данных и внешних сервисов.

Для описания алгоритмов использовались блок-схемы. Блок-схема представляет собой графическое описание структуры алгоритма с помощью стандартных геометрических фигур и линий связи, показывающих порядок выполнения отдельных действий и условий. Такой способ описания делает логику работы модулей наглядной и позволяет отразить как основные сценарии, так и возможные ошибки выполнения.

Алгоритмы информационной системы Media Quality Control Platform представлены модульно. В данном разделе рассматриваются алгоритм регистрации пользователя, алгоритм авторизации пользователя, алгоритм загрузки и анализа медиафайла, а также алгоритм создания нового профиля валидации.

Алгоритм регистрации пользователя в системе представлен на рисунке ...

Последовательность действий алгоритма регистрации пользователя такова:

1. Пользователь переходит к началу выполнения алгоритма регистрации.
2. Система отображает форму регистрации пользователя.
3. Пользователь вводит адрес электронной почты, имя пользователя, ФИО и пароль.
4. Система проверяет, заполнены ли данные регистрационной формы.
5. Если данные не заполнены, пользователь возвращается к вводу адреса электронной почты, имени пользователя, ФИО и пароля.
6. Если данные заполнены, клиентская часть отправляет регистрационные данные на сервер.
7. Серверная часть проверяет, свободен ли указанный адрес электронной почты.
8. Если адрес электронной почты уже используется, система возвращает ошибку регистрации, и пользователь возвращается к вводу данных.
9. Если адрес электронной почты свободен, серверная часть проверяет, свободно ли указанное имя пользователя.
10. Если имя пользователя уже используется, система возвращает ошибку регистрации, и пользователь возвращается к вводу данных.
11. Если имя пользователя свободно, серверная часть выполняет хэширование пароля.
12. Система создает запись данных пользователя в базе данных.
13. Система создает токен подтверждения электронной почты.
14. Система создает запись `REGISTER` в журнале аудита.
15. Система отправляет пользователю уведомление для подтверждения электронной почты.
16. Серверная часть выполняет генерацию JWT-токена.
17. Система возвращает токен и данные пользователя.
18. Система отображает сообщение об успешной авторизации после регистрации.
19. Выполнение алгоритма регистрации завершается.

Алгоритм авторизации пользователя в системе представлен на рисунке ...

Последовательность действий алгоритма авторизации пользователя такова:

1. Пользователь переходит к началу выполнения алгоритма авторизации.
2. Система проверяет, пройдена ли авторизация пользователя.
3. Если пользователь уже авторизован, выполнение алгоритма завершается.
4. Если пользователь не авторизован, система отображает форму ввода данных для авторизации.
5. Пользователь вводит адрес электронной почты и пароль.
6. Система выполняет поиск пользователя по адресу электронной почты.
7. Система проверяет, найден ли пользователь в базе данных.
8. Если пользователь не найден, система возвращает ошибку авторизации, и пользователь возвращается к вводу данных.
9. Если пользователь найден, система проверяет активность учетной записи пользователя.
10. Если учетная запись пользователя неактивна, система выполняет временную блокировку входа.
11. После временной блокировки входа выполнение алгоритма завершается.
12. Если учетная запись пользователя активна, система проверяет количество неудачных попыток входа.
13. Если количество попыток входа меньше допустимого лимита, система выполняет сравнение введенного пароля с хэшем пароля, сохраненным в базе данных.
14. Если количество попыток входа превышает допустимый лимит, система выполняет временную блокировку входа.
15. После временной блокировки входа выполнение алгоритма завершается.
16. Система проверяет, совпадает ли введенный пароль с сохраненным хэшем.
17. Если пароль введен неверно, система записывает неудачную попытку входа в журнал авторизаций.
18. После записи неудачной попытки входа система возвращает ошибку авторизации, и пользователь возвращается к вводу данных.
19. Если пароль введен верно, система записывает успешную попытку входа в журнал авторизаций.
20. Система создает запись о событии `LOGIN` в журнале аудита.
21. Система выполняет генерацию JWT-токена.
22. Система возвращает пользователю токен и данные пользователя.
23. Система отображает сообщение об успешной авторизации.
24. Выполнение алгоритма авторизации завершается.

Алгоритм загрузки и анализа медиафайла представлен на рисунке ...

Последовательность действий алгоритма загрузки и анализа медиафайла такова:

1. Пользователь переходит к началу выполнения алгоритма загрузки и анализа медиафайла.
2. Система выполняет авторизацию пользователя.
3. Система проверяет, авторизован ли пользователь.
4. Если пользователь не авторизован, система возвращает пользователя к этапу авторизации.
5. Если пользователь авторизован, система отображает форму выбора медиафайла или ссылки на файл.
6. Пользователь выбирает медиафайл или указывает ссылку на медиафайл.
7. Система проверяет, выбран ли файл или указана ли ссылка.
8. Если файл или ссылка не выбраны, пользователь возвращается к выбору медиафайла или ссылки.
9. Если файл или ссылка выбраны, пользователь вводит параметры проверки.
10. Система проверяет, заполнены ли параметры проверки.
11. Если параметры проверки не заполнены, пользователь возвращается к вводу параметров.
12. Если параметры проверки заполнены, клиентская часть отправляет данные на сервер.
13. Система проверяет, был ли передан файл или ссылка на медиафайл.
14. Если файл или ссылка не переданы, система возвращает ошибку загрузки.
15. Если файл или ссылка переданы, система выполняет проверку прав доступа пользователя.
16. Если у пользователя нет прав на выполнение операции, система возвращает ошибку доступа.
17. Если права пользователя подтверждены, система получает файл с сервера или по указанной ссылке.
18. Система определяет тип медиафайла.
19. Система выполняет проверку расширения файла.
20. Система проверяет, является ли расширение файла допустимым.
21. Если расширение файла недопустимо, система возвращает ошибку формата файла.
22. Если расширение файла допустимо, система формирует ключ хранения файла.
23. Система загружает файл в объектное хранилище.
24. Система создает или выбирает категорию медиафайла.
25. Система создает или выбирает теги медиафайла.
26. Система рассчитывает контрольную сумму файла.
27. Система создает запись `MediaItem` в базе данных.
28. Система устанавливает для медиафайла статус `UPLOADED`.
29. Система создает запись о событии `UPLOAD_MEDIA` в журнале аудита.
30. Пользователь запускает автоматическую проверку медиафайла.
31. Система передает коды шаблонов проверки `templateCriteriaCodes`.
32. Система выполняет поиск медиафайла по идентификатору.
33. Система проверяет, найден ли медиафайл.
34. Если медиафайл не найден, система возвращает ошибку поиска.
35. Если медиафайл найден, система получает активные критерии проверки.
36. Система выбирает критерии проверки по их кодам.
37. Система проверяет, выбраны ли конкретные критерии.
38. Если конкретные критерии не выбраны, система использует все активные критерии.
39. Если конкретные критерии выбраны, система использует выбранные критерии.
40. Система устанавливает для медиафайла статус `IN_PROCESS`.
41. Система проверяет наличие дубликатов по контрольной сумме файла.
42. Система отправляет данные медиафайла в модуль анализа.
43. Модуль анализа возвращает результат проверки: итоговую оценку, список нарушений и служебные сведения.
44. Система выполняет фильтрацию найденных нарушений.
45. Система пересчитывает оценку качества медиафайла.
46. Система определяет рекомендацию по результатам анализа.
47. Система проверяет, равна ли рекомендация значению `APPROVED`.
48. Если рекомендация равна `APPROVED`, система устанавливает статус `AUTO_CHECKED`.
49. Если рекомендация не равна `APPROVED`, система проверяет, равна ли рекомендация значению `MANUAL_REVIEW`.
50. Если рекомендация равна `MANUAL_REVIEW`, система устанавливает статус `NEEDS_MANUAL_MODERATION`.
51. Если рекомендация не равна `MANUAL_REVIEW`, система устанавливает статус `ON_REVISION`.
52. Система создает запись `QualityCheck` с результатами автоматической проверки.
53. Система сохраняет примененные шаблоны проверки, критерии и найденные нарушения.
54. Система обновляет статус медиафайла в базе данных.
55. Система отправляет уведомление владельцу медиафайла.
56. Система проверяет, требуется ли ручная модерация.
57. Если ручная модерация требуется, система отправляет уведомление модераторам.
58. Если ручная модерация не требуется, система переходит к записи события проверки.
59. Система создает запись о событии `RUN_AUTO_CHECK` в журнале аудита.
60. Система возвращает результат проверки пользователю.
61. Выполнение алгоритма загрузки и анализа медиафайла завершается.

Алгоритм создания нового профиля валидации представлен на рисунке ...

В разрабатываемой системе профиль валидации представлен набором правил качества. Правила качества хранятся в базе данных как записи `QualityRule`. Для профилей проверки используется тип правила `CRITERION`, который описывает критерий автоматической проверки, его код, название, описание, вес и признак активности.

Последовательность действий алгоритма создания нового профиля валидации такова:

1. Администратор переходит к началу выполнения алгоритма создания нового профиля валидации.
2. Система выполняет авторизацию администратора.
3. Администратор переходит в раздел управления профилями валидации.
4. Система отображает форму ввода параметров конфигурации профиля.
5. Администратор вводит параметры конфигурации профиля валидации.
6. Система проверяет, заполнены ли параметры конфигурации.
7. Если параметры конфигурации не заполнены, администратор возвращается к вводу параметров.
8. Если параметры конфигурации заполнены, система переходит к созданию нового профиля валидации.
9. Администратор выбирает тип медиафайла: `IMAGE`, `VIDEO`, `AUDIO`, `TEXT` или `MIXED`.
10. Система проверяет, выбран ли шаблон профиля валидации.
11. Если шаблон выбран, система загружает критерии выбранного шаблона.
12. Если шаблон не выбран, система создает пользовательский профиль типа `custom`.
13. Система отображает список критериев проверки.
14. Администратор выбирает критерии проверки для профиля валидации.
15. Система проверяет, выбраны ли критерии проверки.
16. Если критерии проверки не выбраны, система отображает ошибку с требованием выбрать критерии.
17. После отображения ошибки администратор возвращается к выбору критериев проверки.
18. Если критерии проверки выбраны, система переходит к настройке параметров критериев.
19. Администратор настраивает параметры выбранных критериев.
20. Система проверяет валидность введенных данных.
21. Если данные невалидны, система отображает ошибку валидации данных.
22. После отображения ошибки администратор возвращается к настройке профиля валидации.
23. Если данные валидны, система проверяет, существует ли профиль валидации с такими параметрами.
24. Если профиль уже существует, система обновляет существующий профиль.
25. Если профиль не существует, система создает новый профиль.
26. Система сохраняет критерии проверки и словарь нарушений.
27. Система создает запись о событии в журнале аудита.
28. Система фиксирует, что профиль валидации сохранен.
29. Система отображает сохраненный профиль в интерфейсе.
30. Выполнение алгоритма создания нового профиля валидации завершается.
